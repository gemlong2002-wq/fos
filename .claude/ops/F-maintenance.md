# F. 維護協議：弱模型如何安全更新制度檔

> 制度檔 = CLAUDE.md + `ops/` 下所有檔案。本協議規定誰能改什麼、怎麼改。

## 1. 權限分級

| 級別 | 內容 | 規則 |
|---|---|---|
| 可自行改 | `ops/lessons.md` 追加教訓；修正明確的事實錯誤（路徑打錯、指令失效）；範例的補充 | 改前備份，改後 read-back，回報中告知使用者「已修正 X」 |
| 先提案再改 | 調整判準的門檻數值（如重試 2 輪改 3 輪）；新增規則；把教訓升格為正式規則 | 寫出提案（現狀/問題/改法/風險），問使用者，核准才動 |
| 不可改 | CLAUDE.md 鐵律區；`C-dispatch.md` 的升降級路徑與「驗證不自驗」；本檔權限分級表 | 認為有問題 → 記入 lessons.md 並向使用者提出，由使用者親自改 |

判定不確定落在哪級 → 按較嚴一級處理。

## 2. 修改程序（所有級別通用）

1. `cp 目標檔 目標檔.bak-YYYYMMDD`（同目錄；已有當日備份則不覆蓋）。
2. 用局部 edit，不整檔重寫。
3. read-back 改動處，並 grep 相關關鍵字確認沒有與其他檔規則衝突。
4. 在回報中一句話說明：改了哪檔哪條、為什麼。

## 3. 教訓登記（lessons.md）

**時機**：任務中出現「重試 ≥2 輪」「驗收 FAIL」「使用者糾正」任一，收尾時必須登記。
**格式**（一行一條，追加到檔尾）：

```
- [YYYY-MM-DD][類別:調度|驗收|工具|環境|其他] 症狀一句話 → 教訓一句話（可執行形式）
```

範例：
```
- [2026-07-10][工具] pytest 在 repo 根目錄跑會漏掉 plugins/ 的測試 → 一律用 `pytest -c pyproject.toml` 從根目錄跑
- [2026-07-12][調度] Haiku 做正則替換把註解裡的字串也改了 → 批次替換的驗收條件必須含「diff 逐檔抽查 3 處」
```

**使用**：主對話在開始高風險任務前，掃 lessons.md 中同類別條目。

## 4. 精簡週期（防止制度膨脹成沒人讀的法典）

觸發任一即執行精簡（提案級操作，需使用者核准後動手）：
- `lessons.md` 超過 40 條，或任一制度檔超過 250 行。
- 發現 ≥3 條教訓指向同一根因。

精簡做法：
- 同根因教訓合併成一條正式規則，提案寫入對應制度檔；原教訓移入 `ops/graveyard.md` 附日期與去向。
- 6 個月內未被引用過的規則 → 提案移入 graveyard（移入即註記日期，graveyard 只增不刪）。
- 精簡的驗收：由 fresh-context agent 比對精簡前後，確認沒有語意遺失，只有壓縮。

## 5. 版本紀錄

每個制度檔檔尾維護一個 changelog 區塊，每次修改追加一行：`YYYY-MM-DD 誰(模型) 改了什麼一句話`。這是弱模型之間唯一的跨 session 修改審計線。
