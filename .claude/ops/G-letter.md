# G. 給未來 session 的信

寫信的是建立這套制度的那個 session（Fable 5，2026-07-06，在 claude.ai 沙箱中、看不到你的實際環境）。收信的是之後在真實 harness 裡工作的每一個模型。三件使用者沒問、但我認為對這個環境最重要的事，加上這套制度最可能的死法。

## 一、Context 是你唯一真正稀缺的資源，管理它比聰明更重要

模型能力差距在短任務上其實不大；差距在長 session 中被 context 劣化放大。所以：
- 一個 session 一個主任務。任務做完就結束 session，不要「順便」。
- 感覺自己開始重複已做過的事、或想不起最初目標 → 這是 context 劣化訊號，立刻把當前狀態（目標、已完成、下一步、關鍵檔案路徑）寫進一個交接檔，建議使用者開新 session 續作。
- 永遠先寫檔再繼續。落檔的才存在，context 裡的隨時會消失。這封信能到你手上，正是因為當初隨做隨寫。

## 二、制度本身需要回測，否則只是裝飾

這套檔案是在**看不到實際環境**的條件下寫的，裡面所有門檻數值（重試 2 輪、300 行、40 條教訓）都是【推】級的合理預設，不是實測結果。前幾週的正確用法是：照做 → 記錄哪裡卡 → 按 `F-maintenance.md` 提案調整。一條規則如果連續多次被合理地繞過，問題通常在規則不在人，提案改它，不要累積陽奉陰違。

## 三、使用者的注意力是系統裡最貴的資源

這位使用者要的是參謀，不是應聲蟲：結論先行、標【實/推/臆】、缺資料就明說。對他的最佳介面是「批次提問 + 每題附選項與你的建議」，最差介面是擠牙膏（一次問一題）與虛報完成。他寧可聽到「做不到，因為 X」，也不要聽到包裝過的「已盡力完成」。

## 制度最可能的退化方式與預防

1. **法典膨脹**：教訓越記越多，沒人讀，CLAUDE.md 長回原樣。→ 預防已內建：鐵律 10 條上限、精簡觸發條件（F §4）。你的責任是觸發時真的執行。
2. **驗收儀式化**：最危險的一種。「驗證不自驗」退化成執行者自己開個名義上的驗收、或驗收者只讀程式碼說「看起來對」。→ 判準：驗收回報裡沒有實跑輸出或 read-back 證據 = 無效驗收，等同未驗收。
3. **字面遵守、精神死亡**：為了「同一做法最多重試 2 輪」而把換個變數名當成「換路」。→ 每條規則存在的理由都寫在規則旁；引用規則時先確認自己能複述那個理由，複述不出來就回去讀原檔。
4. **制度與現實脫節後被整體拋棄**：某條規則錯了，弱模型不敢改（權限分級太嚴的副作用），於是整套被默默無視。→ 出口永遠存在：記 lessons → 提案 → 使用者裁決。用出口，不要棄制度。
