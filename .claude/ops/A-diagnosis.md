# A. Harness 快速診斷：前三名漏洞與修法

> 標記：全篇【推】。依據是 agent harness 的通用失效模式，非你的實際 transcript。
> 升【實】方法：提供 (a) 現有 CLAUDE.md、(b) 2–3 份出過問題的 session transcript。
> 第一個安裝 session 應執行 `00-INSTALL.md` 的校準步驟，用實際資料驗證或修正本診斷。

## 第 1 名：指揮官下場（最漏 token、最容易失焦）

**症狀**：主對話直接大量讀檔、掃 repo、抓網頁、貼整份工具輸出進 context。
**後果**：
- Context 被原始資料塞滿，任務目標被推出視野 → 中後段失焦、遺忘早期指令。
- 觸發 compaction 後細節遺失，模型開始「重新發現」已知事實，重複工作。
- 一次大掃描可燒掉整個 session 一半以上的 context【推】。

**修法**（詳見 `C-dispatch.md`）：
- 鐵律：主對話不執行「大量讀取、全 repo 掃描、網頁抓取、批次改檔」，一律派 subagent。
- 回報合約：subagent 只回結論 + 檔案:行號；長產物落檔傳路徑，不貼全文。
- 量化判準：預估要讀超過 3 個檔或 300 行 → 派工，不自己讀。

## 第 2 名：無驗收條件的派工 + 自驗（最容易出錯）

**症狀**：派工只說「去做 X」，沒寫驗收條件；做完由做的那個模型自己宣稱「完成」。
**後果**：
- Subagent 對「完成」自由心證：改了但沒測、測了但只測 happy path、聲稱寫了檔其實沒落地。
- 自驗有確認偏誤：寫程式的 context 裡「這段是對的」已成既定事實，驗不出自己的錯。
- 錯誤沉默通過，到很後面才爆，回溯成本極高。

**修法**（詳見 `C-dispatch.md` §驗證、`E-delegation-templates.md`）:
- 派工三件套強制：目標與動機、可機械判定的驗收條件、回報格式。
- 驗證不自驗：驗收一律派 fresh-context agent。檔案用 read-back、程式碼用測試或實跑、高風險判斷加第二意見。

## 第 3 名：同路重試迴圈（漏 token + 出錯的複合型）

**症狀**：一個做法失敗後，用幾乎相同的方式再試第二、三、四次；或小模型卡住後主模型不升級、繼續讓它撞牆。
**後果**：
- 每輪重試都帶著前幾輪的失敗殘骸，context 越來越髒，成功率反而下降。
- 燒掉的 token 常超過「直接升級模型一次做對」的成本。

**修法**（詳見 `C-dispatch.md` §升降級、`D-judgment.md` §換路訊號）：
- 同一做法最多重試 2 輪，第 3 次前必須換路或升級。
- Haiku 級錯 1 次直接升 Sonnet；Sonnet 同一子任務連錯 2 次，帶完整失敗軌跡升 Opus。
- 「換路」的定義：改變方法或假設，不是換措辭重跑同一件事。

## 次要但常見（列出供校準時比對）

- 局部修改卻整檔重寫 → 用 edit/diff 工具，驗證只 diff 改動處。
- 測試輸出全量貼回 → 只回 pass/fail 統計 + 失敗案例的 stack trace 前 5 行。
- 每個 session 塞多個不相關任務 → context 汙染；一個 session 一個任務。
